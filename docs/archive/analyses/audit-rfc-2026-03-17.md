# Vue d’ensemble avant le détail

À l’échelle de l’ensemble, la qualité la plus frappante est la suivante : **les RFC ont une colonne vertébrale réelle**. On voit une architecture pensée comme un système complet, avec principes fondateurs, taxonomie claire, invariants, anti-patterns, typage strict, cycle de vie, stratégie de rendu, et séparation forte entre métier, state, communication et UI.

Ma lecture synthétique est donc :

- **RFC-0001** = excellente RFC de fondation architecturale, très solide sur les frontières et la philosophie, mais encore un peu inégale sur le niveau de normativité de certaines sections.
- **RFC-0002 (index / API / typage)** = très bonne RFC de traduction technique, avec un vrai niveau de sophistication TypeScript, mais encore quelques zones à verrouiller pour passer de “vision robuste” à “contrat d’implémentation sans ambiguïté”.
- **Feature / Entity / Channel** = trio très cohérent, bien articulé, avec de bonnes décisions structurantes ; les risques sont davantage dans l’**implémentation runtime** que dans le **modèle conceptuel**.

# Audit — RFC-0001 : Architecture fondamentale

## Ce que la RFC réussit très bien

Cette RFC pose une **architecture front-end à identité forte**, ce qui est rare. Elle ne se contente pas de dire “on aura des composants” : elle distingue explicitement une **couche abstraite persistante** (Application, Feature, Entity, Channel, Router, Radio) et une **couche concrète éphémère** (View, Behavior, Foundation, Composer), puis relie cette taxonomie à des responsabilités et à un cycle de vie. C’est une vraie force de conception.

La partie la plus solide, à mon sens, est la formalisation des **principes fondateurs** : framework-first, flux unidirectionnel strict, dépendances déclaratives, encapsulation stricte du state, testabilité, lisibilité, typage fort. Ce sont de vrais principes architecturaux, pas des slogans, et ils irriguent ensuite les invariants et les décisions.

Autre point très fort : le traitement des **invariants**. Le fait d’avoir une liste I1–I41, couvrant droits de communication, ownership du state, cycle de vie, altération DOM, déclaration des dépendances, etc., donne au framework une base presque “constitutionnelle”. C’est excellent pour éviter les dérives futures.

Enfin, la RFC assume un vrai parti pris sur la **Feature comme composant métier unique** avec 5 capacités (emit, handle, listen, reply, request), sans multiplication artificielle de sous-types métier. C’est une décision très saine : simple, lisible, forte, et surtout cohérente avec l’architecture chorégraphique choisie.

## Là où je vois les faiblesses

Le principal défaut de cette RFC n’est pas conceptuel, il est **éditorial / normatif** : certaines sections sont beaucoup plus abouties que d’autres. Les parties sur la View, la Foundation, le Composer, les droits DOM et la matrice de communication sont déjà très structurées ; d’autres rubriques sont encore plus “intentionnelles” que normatives.

En particulier, la RFC décrit très bien le **quoi** et le **pourquoi**, mais parfois un peu moins le **niveau de contrainte exact** attendu pour l’implémentation. Pour une RFC d’architecture, ce n’est pas dramatique, mais cela veut dire qu’elle fonctionne très bien comme **document de direction**, un peu moins comme **base de validation technique exhaustive** sans RFC complémentaires.

Je note aussi un risque de **sur-prescription** : le framework interdit énormément de choses (state local UI, accès direct DOM, composition portée par la View, cross-domain trigger/emit, etc.). C’est cohérent, mais cela suppose que tu documentes ensuite très bien les **cas limites d’usage réel**, sinon la doctrine peut paraître “parfaite sur le papier, dure à vivre dans le code”.

## Verdict sur RFC-0001

Je considère cette RFC comme **très bonne** et même **remarquablement structurante**. Pour moi, elle est **suffisamment robuste pour servir de constitution architecturale**, mais pas encore complètement “fermée” au sens où toute implémentation tierce produirait exactement le même framework sans interprétation.

### Note
- **Solidité architecturale** : **9,5/10**
- **Normativité exploitable telle quelle** : **8/10**

### Recommandations prioritaires
1. Transformer explicitement certaines sections encore descriptives en **règles normatives** (“DOIT / NE DOIT PAS / PEUT”).
2. Ajouter un **glossaire des cas limites** : “comment traiter un slider”, “un état UI transitoire”, “une modale”, “une animation”, etc., pour montrer que la doctrine tient dans les usages concrets.
3. Ajouter une courte section “**ce que Bonsai optimise** / **ce que Bonsai n’optimise pas**” pour assumer son positionnement opinionated.

# Audit — RFC-0002 : API et contrats de typage

## Ce que la RFC réussit très bien

Cette RFC fait ce qu’une bonne RFC technique doit faire : elle **traduit l’architecture en contrats exploitables**. Elle explicite que RFC‑0001 décrit le “quoi”, et qu’ici on décrit le “comment” : signatures TypeScript, mécanismes de typage, conventions `onXXX`, cycle de vie, validation compile-time et runtime. C’est très propre comme articulation documentaire.

La qualité majeure de cette RFC, c’est la maîtrise du **TypeScript comme langage de conception**, pas seulement comme langage d’implémentation. L’usage exclusif de `type`, les génériques contraints, les template literal types, les mapped types, les conditional types, et l’idée “types d’abord, récompense ensuite” montrent un cadre très mature.

J’apprécie aussi beaucoup le fait que le document serve d’**index transversal** : il ne noie pas tout dans un seul monolithe, mais garde les conventions, la vision globale, les validations et les mécanismes communs, tout en déléguant les détails de Feature / Entity / Channel aux documents dédiés. C’est une bonne stratégie documentaire.

Autre force : la RFC ne se contente pas de typer les composants ; elle type aussi la **discipline d’usage**. En d’autres termes, le type system n’est pas là juste pour sécuriser des payloads, il sert à matérialiser les **droits architecturaux**. C’est excellent.

## Là où je vois les faiblesses

La plus grosse limite de cette RFC, c’est qu’elle est **très ambitieuse** — ce qui est une qualité — mais du coup elle agrège aussi beaucoup de sujets critiques : validation compile-time, validation runtime, cycle de vie, rendu View, metas, Radio, bootstrap, Foundation, Composer. Cela la rend extrêmement riche, mais également **plus difficile à stabiliser**.

Ensuite, on sent encore quelques zones où la spec dit “voilà l’intention” plus que “voilà la règle finale”. C’est particulièrement sensible sur :

- l’injection des **metas** et leur contexte causal ;
- la **validation runtime** en mode debug / strict / production ;
- certaines parties de bootstrap et de diagnostics ;
- la sémantique exacte de plusieurs garde-fous.

Enfin, la partie View / PDR est intellectuellement très intéressante, mais elle mériterait presque d’être isolée dans une RFC dédiée tant elle est riche et structurante. En l’état, elle est bien présente, mais elle alourdit beaucoup RFC‑0002.

## Verdict sur RFC-0002

C’est une **très bonne RFC de traduction technique**. Elle donne au projet son épaisseur d’ingénierie. Sans elle, Bonsai resterait une architecture élégante ; avec elle, il devient un framework potentiellement implémentable.

Mais je la vois encore comme une RFC **de très bon niveau draft**, pas comme une RFC “gelée”. Il reste quelques sujets à transformer en **contrats réellement normatifs** ou en RFC spécialisées.

### Note
- **Qualité technique / typage** : **9,5/10**
- **Complétude normative pour implémentation v1** : **7,5/10**

### Recommandations prioritaires
1. Extraire ou renforcer une **RFC dédiée au runtime/validation** (erreurs, diagnostics, strict/debug/prod).
2. Figer formellement le **contrat de metas** : création, propagation, contexte async, erreurs.
3. Envisager une RFC séparée “**Rendering / PDR / Templates / DOM scope**” pour alléger RFC‑0002 et clarifier la responsabilité documentaire.

# Audit — RFC-0002 Feature

## Ce que la RFC réussit très bien

Le document Feature est probablement celui qui exprime le mieux la philosophie de Bonsai : une **classe abstraite métier**, paramétrée par la structure Entity et le Channel, avec 5 capacités clairement définies, des handlers auto-découverts, un cycle de vie simple, et une relation 1:1 avec son Entity.

Le point fort majeur est le couple :

- **déclarations statiques** (`listen`, `request`, namespace) ;
- **handlers générés / vérifiés via TypeScript**.

C’est une très bonne synthèse entre **déclaratif** et **conventionnel** : le développeur déclare les dépendances une fois, et le framework + le compilateur sécurisent le reste.

J’aime aussi beaucoup la clarté des 5 capacités. Beaucoup de frameworks mélangent trop facilement “commande”, “événement”, “lecture”, “effet”, “service”. Ici, les capacités sont nettes, sémantiques, et cohérentes avec le tri-lane Channel.

## Là où je vois les faiblesses

Le principal risque ici est l’**auto-discovery `onXXX`**. Conceptuellement, c’est très élégant. Industriellement, cela suppose une discipline forte et une implémentation très solide côté runtime/bootstrap. Le moindre flou sur la découverte, les collisions, les handlers orphelins, ou les erreurs de nommage risque de coûter cher en DX.

Le second point à verrouiller est le **contrat d’erreur** : que se passe-t-il si un handler command throw ? Si un request handler rejette ? Si un event listener échoue au milieu d’une cascade ? Le document pose bien les capacités, mais pas encore toute la jurisprudence d’exécution.

Le cycle de vie `onInit()` / `onDestroy()` est volontairement simple, ce qui est très bien. En revanche, le choix d’un bootstrap séquentiel pour les `onInit()` async devra être confirmé ou modulé selon les besoins de startup.

## Verdict sur RFC Feature

Très bonne RFC. C’est une des plus convaincantes du lot. Elle est **forte sur le design**, **claire sur la responsabilité**, **solide sur le typage**, et ne souffre que de quelques manques d’implémentation normative autour du runtime et des erreurs.

### Note
- **Clarté du contrat Feature** : **9/10**
- **Risque d’implémentation** : **modéré**

### Recommandations prioritaires
1. Formaliser une matrice “**si handler X échoue, alors comportement Y**”.
2. Spécifier le traitement des **handlers fantômes**, **non déclarés**, **manquants**, **dupliqués** au bootstrap.
3. Écrire quelques **exemples normatifs multi-Features** pour valider la chorégraphie réelle.

# Audit — RFC-0002 Entity

## Ce que la RFC réussit très bien

Cette RFC est très forte sur l’**encapsulation du state**. Elle définit clairement :

- un `TEntityStructure` jsonifiable ;
- un `state` protégé comme source unique de vérité ;
- des méthodes de mutation explicites ;
- des méthodes de query ;
- une notification interne Entity → Feature ;
- `toJSON` / `fromJSON` pour snapshot/restauration.

Le choix “**JsonSerializable obligatoire**” est excellent pour un framework qui veut rendre possible persistance, time-travel, snapshots, diff et robustesse de contrat. C’est un choix dur, mais très cohérent.

L’idée des handlers `on<Key>EntityUpdated` et `onAnyEntityUpdated`, inspirée des patterns de Backbone/Marionette mais typée et réinterprétée, est particulièrement bonne. C’est une vraie plus-value : tu donnes à la Feature un moyen de réagir au **changement d’état réel** sans l’obliger à tout dupliquer au niveau des command handlers.

## Là où je vois les faiblesses

Le principal risque est de loin le **coût et la complexité de la mécanique de notification**. Le document décrit un modèle avec snapshots avant/après, diff, changedKeys, handlers per-key, catch-all, ré-entrance, file FIFO, profondeur maximale. C’est conceptuellement très fort, mais c’est aussi un des coins les plus compliqués du runtime.

En d’autres termes : le **contrat est bon**, mais il faudra faire extrêmement attention à la mise en œuvre :

- coût mémoire des snapshots,
- coût CPU du diff,
- comportement sous mutations fréquentes,
- cohérence en asynchrone,
- ergonomie debug.

Deuxième point : la frontière entre **logique métier** (Feature) et **logique de données** (Entity) est bien posée, mais elle devra être illustrée très concrètement dans la doc d’usage, sinon certaines équipes risquent de dériver soit vers des Entities trop riches, soit vers des Features trop lourdes.

## Verdict sur RFC Entity

Pour moi, c’est une RFC **très intelligente**, probablement l’une des plus originales de l’ensemble. Elle donne à Bonsai une vraie identité de gestion d’état. Mais c’est aussi celle qui porte l’un des **plus gros risques de complexité runtime**.

### Note
- **Design conceptuel** : **9/10**
- **Risque runtime / perf** : **élevé**

### Recommandations prioritaires
1. Écrire un **ADR d’implémentation runtime Entity** (snapshot/diff/proxy/patches/dirty flags).
2. Formaliser des **budgets de perf** ou au moins des cibles qualitatives.
3. Ajouter une **doc de design heuristics** : qu’est-ce qui va dans Entity, qu’est-ce qui reste dans Feature.

# Audit — RFC-0002 Channel

## Ce que la RFC réussit très bien

C’est une RFC courte mais très structurante. Elle formalise très bien :

- la distinction entre **contrat de typage** (`TChannelDefinition`) et **instance runtime interne** ;
- le modèle **tri-lane** ;
- le pattern **namespace TypeScript** ;
- la co-localisation avec la Feature ;
- le fait que le runtime Channel est un détail d’implémentation, jamais exposé.

Le design `declareChannel<T>()` est élégant : le développeur manipule un token léger à runtime mais riche au niveau type. C’est propre, simple et très bien aligné avec le reste de l’architecture.

Le tri-lane est lui aussi très solide. C’est probablement l’une des meilleures décisions du framework, car elle force une grammaire claire de communication entre composants.

## Là où je vois les faiblesses

La faiblesse n’est pas dans le contrat, mais dans les **implications runtime non totalement closes**. La RFC dit que l’implémentation peut s’appuyer sur rxjs, mais que ce n’est qu’un détail interne. Très bien. En revanche, les comportements exacts à formaliser restent encore, notamment :

- absence de handler/replier,
- politique d’erreur sur `dispatchRequest`,
- garanties d’ordre,
- teardown / unsubscribe / fuites,
- interaction précise avec metas et hop.

Autre point : le pattern `namespace` TypeScript est puissant, mais il mérite d’être validé contre les contraintes réelles d’outillage, bundling, tree-shaking, ergonomie IDE et conventions d’import. Conceptuellement, c’est très bon ; techniquement, il faudra le tester dans de vrais contextes de build.

## Verdict sur RFC Channel

Très bonne RFC, conceptuellement propre, probablement la plus “compacte et nette” du lot. Elle expose bien la distinction **contrat public / runtime privé**, ce qui est une très bonne propriété d’architecture.

### Note
- **Clarté du contrat** : **9/10**
- **Dépendance à des choix runtime encore à figer** : **moyenne**

### Recommandations prioritaires
1. Rédiger une courte **RFC ou ADR “Channel runtime semantics”**.
2. Figer les comportements “**no handler / no replier / teardown / error propagation**”.
3. Tester très tôt le pattern `namespace` dans le pipeline réel (tsc, bundler, IDE, extraction types).

# Diagnostic transversal : ce qui est déjà prêt, ce qui ne l’est pas

## Ce que je considère comme déjà très mûr

Je considère comme **très mûrs** les points suivants :

- la philosophie générale du framework ;
- la séparation couche abstraite / couche concrète ;
- le modèle Feature / Entity / Channel ;
- la discipline de communication tri-lane ;
- l’approche TypeScript-first ;
- l’interdiction du state local UI comme doctrine centrale.

## Ce qui, selon moi, n’est pas encore totalement “v1-ready”

Je considère encore comme **à verrouiller** :

- la sémantique runtime des erreurs et des garde-fous ;
- la mécanique exacte des metas ;
- la stratégie d’implémentation Entity côté diff/notifications/perf ;
- la validation runtime / mode debug / strict / production ;
- la stabilisation documentaire de la partie PDR/rendu comme domaine propre.

---

# Mon verdict final, RFC par RFC

- **RFC-0001** : **très robuste**, excellent socle architectural, à renforcer surtout sur la normativité de certaines zones.
- **RFC-0002** : **très forte techniquement**, mais encore en phase de consolidation documentaire avant gel complet.
- **RFC Feature** : **très convaincante**, proche d’un contrat exploitable, manque surtout la jurisprudence runtime.
- **RFC Entity** : **conceptuellement excellente**, mais c’est la zone la plus risquée côté implémentation et performance.
- **RFC Channel** : **très propre et très cohérente**, à compléter par une sémantique runtime plus fermée.

# Pistes de suite possibles

Si besoin, la suite la plus utile peut être l’un des trois livrables suivants :

## Option A — plan d’action concret
Un **backlog de durcissement RFC** avec :
- priorité P1 / P2 / P3,
- ce qu’il faut transformer en ADR,
- ce qu’il faut déplacer en RFC dédiée,
- et les dépendances entre documents.

## Option B — audit normatif
Une reprise de chaque RFC avec :
- formulations trop floues,
- phrases à réécrire en “DOIT / NE DOIT PAS / PEUT”,
- règles implicites à rendre explicites.

## Option C — préparation implémentation
Une **checklist “avant de coder Bonsai v1”** couvrant :
- runtime,
- performance,
- tests de conformité,
- outillage TypeScript,
- stratégie de debug.
